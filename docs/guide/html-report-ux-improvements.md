# HTML Report UX Improvements

**Status:** Proposed
**Informed by:** Real-world usage review of reports hosted at `reports.hotspots.dev`
**Relevant source:** `hotspots-core/src/html.rs`

This document tracks specific, scoped improvements to the HTML report's usability for readers who are unfamiliar with hotspots or code quality metrics. These are not cosmetic changes — each addresses a concrete gap between what the report shows and what a user needs to act on it.

---

## Problems identified

### P1 — Metric abbreviations have no visible glossary

**Current state:** CC, ND, FO, LRS appear as column headers. Tooltips (`title` attributes) exist on most headers but require hover to discover. A reader scanning the page has no way to know what these mean.

**Impact:** Developers unfamiliar with cyclomatic complexity or fan-out cannot self-serve. The report depends on the user already knowing the methodology.

**Files:** `html.rs` → `render_functions_table()` around line 1476.

---

### P2 — Band thresholds are unstated

**Current state:** Functions are labeled `critical / high / moderate / low` but the report never shows what LRS values correspond to which band. The user sees a red badge but doesn't know whether "critical" means LRS > 10, > 20, or something relative.

**Impact:** Users cannot judge severity calibration or compare across repos without context.

**Files:** `html.rs` → `render_summary()`, `render_triage_panel()`.

---

### P3 — Activity Risk formula is opaque

**Current state:** The summary shows "Total Activity Risk" as a number (e.g. `9,363.9`). The triage table shows per-function activity risk scores. Neither explains that these are `LRS × recent_commit_frequency` (approximately), or what a "high" vs "low" activity risk score means.

**Impact:** The core differentiator of hotspots — activity-weighting — is invisible to a new reader. They see a complexity tool, not an activity-weighted risk tool.

**Files:** `html.rs` → `render_summary()`, triage table header.

---

### P4 — Triage quadrant zero-states are misleading

**Current state:** The triage quadrant shows four buckets: Active Risk, Stable Debt, Watch, OK. For a repo like hexo with 44 critical functions but few recent changes, "Active Risk: 0" is displayed prominently. This looks like "nothing to do right now" when 44 critical-band functions exist — they're just in "Stable Debt."

**Impact:** Users may interpret "0 active risks" as a clean bill of health and miss the stable debt.

**Proposed fix:** When "Active Risk" is 0 but "Stable Debt" is non-zero, add a note: _"No critical functions with recent activity — but X stable-debt functions are awaiting a refactor sprint."_ This is one sentence, not a redesign.

**Files:** `html.rs` → `render_triage_panel()` around line 1668.

---

### P5 — No back-link to blog analysis post

**Current state:** Reports are self-contained HTML files. The `render_header()` and `render_footer()` functions have no knowledge of a corresponding blog post URL. When reports are hosted at `reports.hotspots.dev`, users who land on a report directly have no path to the written analysis (which contains function-specific recommendations, context, and interpretation).

**Proposed fix:** Accept an optional `source_url: Option<&str>` parameter in `render_html_snapshot()`. When set, add a banner below the header:

```html
<div class="source-banner">
  This report is accompanied by a written analysis: <a href="{url}">Read the full analysis →</a>
</div>
```

When not set (local CLI use, CI use), render nothing. The upload pipeline in `hotspots-cloud` can pass the blog post URL at upload time.

**Files:**
- `html.rs` → `render_html_snapshot()` signature + `render_header()` or new `render_source_banner()`
- `hotspots-cloud/scripts/upload_report.py` (or equivalent) passes the post URL

---

### P6 — Footer is a dead end

**Current state:** `render_footer()` returns `"Generated by Hotspots"` with no links.

**Proposed fix:** Add links to:
- The hotspots GitHub repo (`github.com/Stephen-Collins-tech/hotspots`)
- The metrics reference doc (`hotspots.dev/docs/reference/metrics`)
- A one-line explanation of the methodology

Example:
```html
<footer>
  <p>
    Generated by <a href="https://github.com/Stephen-Collins-tech/hotspots">hotspots</a> —
    activity-weighted code risk analysis.
    <a href="https://hotspots.dev/docs/reference/metrics">How metrics are calculated →</a>
  </p>
</footer>
```

**Files:** `html.rs` → `render_footer()` at line 2316.

## Implementation notes

### Metric glossary (P1)

Option A — **Collapsible panel above the functions table.** Renders once, can be dismissed. Low visual noise.

Option B — **Inline tooltip expansion on `?` icon next to each column header.** Lower page weight but requires JS click handler; hover-only is already there and not sufficient.

Recommendation: Option A. A single collapsible `<details>` element above the table with a one-line definition per metric. No JS required. Example:

```html
<details class="metric-glossary">
  <summary>Metric definitions</summary>
  <dl>
    <dt>LRS</dt><dd>Local Risk Score — composite of CC, ND, FO, and NS. Higher = harder to change safely.</dd>
    <dt>CC</dt><dd>Cyclomatic Complexity — number of independent execution paths. Each path is a required test case.</dd>
    <dt>ND</dt><dd>Nesting Depth — maximum level of nested control structures (if/for/while/try). ≥5 is deep.</dd>
    <dt>FO</dt><dd>Fan-out — number of distinct functions this function calls. High fan-out = broad coupling footprint.</dd>
    <dt>NS</dt><dd>Nesting Statements — total count of control-flow statements. Drives exit_heavy and long_function patterns.</dd>
    <dt>Activity Risk</dt><dd>LRS × recent commit frequency (weighted by recency). The core hotspots signal: complexity × churn.</dd>
  </dl>
</details>
```

### Band threshold legend (P2)

Add to the summary section or as a footnote to the functions table:

```html
<p class="band-legend">
  Bands: <span class="band-critical">critical</span> LRS ≥ 9 ·
         <span class="band-high">high</span> LRS 6–8.9 ·
         <span class="band-moderate">moderate</span> LRS 3–5.9 ·
         <span class="band-low">low</span> LRS < 3
</p>
```

Thresholds sourced from `hotspots-core/src/risk.rs` → `RiskThresholds::default()`: `moderate: 3.0, high: 6.0, critical: 9.0`. These are overridable via `.hotspotsrc.json`; the legend should ideally reflect the resolved thresholds, not hardcoded defaults — but static defaults are correct for the public hosted reports which use default config.
